## 一、产品概述

### 1.1 产品名称

**老母记账** — 一款轻量、好用的个人记账应用。

### 1.2 产品定位

面向普通个人和家庭用户的日常记账工具。不需要财务知识，打开就能记，像记流水账一样简单。

### 1.3 目标用户

- 想养成记账习惯但不想用复杂软件的人
- 想清楚知道"钱花哪儿了"的家庭管账人
- 只需要记支出、不需要记收入的人群

### 1.4 运行平台

- **Windows 电脑**（桌面应用）
- **苹果手机 iPhone**（移动应用）

## 二、功能需求

### 2.1 核心功能：记录支出 ✅ 必须实现

| 功能点 | 说明 |
|--------|------|
| 记一笔支出 | 输入金额（人民币 ¥）、选择分类、自动记录日期，可选填备注 |
| 金额输入 | 支持小数点后两位（精确到分），如 12.50 元 |
| 分类选择 | 从预设的 12 个分类中选择一个 |
| 日期记录 | 默认当天，也可以手动改成其他日期 |
| 备注 | 可自由填写的文字备注（选填），比如"超市买菜" |

### 2.2 账单列表 ✅ 必须实现

- 按时间倒序显示所有支出记录（最新的在最上面）
- 每条记录显示：日期、金额、分类、备注摘要
- 支持按分类筛选（只看某一类的花销）
- 支持按日期范围筛选（只看某个月）

### 2.3 月度统计 ✅ 必须实现

- 本月总支出金额
- 各类别支出占比（用简单的图表或百分比展示）
- 每日支出趋势（哪天花得多一清二楚）

### 2.4 数据管理 ✅ 必须实现

- 数据保存在手机/电脑本地，不上传服务器（保护隐私）
- 编辑已记录的账单（改金额、改分类等）
- 删除某条记录
- 数据备份与恢复（可选，后续版本考虑）

### 2.5 后续可扩展功能 ⭐ 以后再加

- 收入记录
- 预算设置与提醒
- 多账户管理（支付宝、微信、银行卡等）
- 数据导出为 Excel
- 云同步（多设备数据同步）

## 四、技术栈方案对比

> ⚠️ **写给技术小白的说明**：  
> "技术栈"就是用来做这个 App 的一套工具和"积木"。不同的选择会影响开发速度、应用体验、以及将来好不好维护。下面我用大白话解释每个方案。

---

### 方案 A：Flutter（⭐ 推荐方案）

**一句话解释**：写一次代码，同时生成 Windows 电脑应用 + 苹果手机应用，就像做一份菜同时装两个盘子。

| | 说明 |
|---|---|
| **用什么语言** | Dart（谷歌设计的一门编程语言） |
| **谁在用** | 谷歌、阿里巴巴、字节跳动、宝马等 |
| **手机上** | 直接生成 iOS 原生应用，上架 App Store |
| **电脑上** | 直接生成 Windows 安装包（.exe） |

**👍 优点（为什么推荐）：**

1. **真正的"写一次，到处跑"**：同一份代码，Windows 和 iPhone 都能用，不需要分开写两套
2. **界面好看且流畅**：自带漂亮的 Material Design 风格组件，滑动、动画都很丝滑，不卡顿
3. **开发快**：改完代码一两秒就能看到效果，不用漫长等待
4. **有大公司撑腰**：Google 维护，不会突然没人管了
5. **社区人多**：遇到问题容易搜到答案，插件和教程也丰富
6. **打包简单**：一键生成 .exe 和 iOS App

**👎 缺点：**

1. **应用体积稍大**：安装包可能比纯原生 App 大几十 MB（但对记账类应用来说不算问题）
2. **语言冷门一些**：Dart 用的人比 JavaScript 少，不过学习成本并不高

---

### 方案 B：网页技术方案（React + Electron / Capacitor）

**一句话解释**：用做网页的技术来做 App，就像把一个网页"包装"成手机应用和电脑应用。

| | 说明 |
|---|---|
| **用什么语言** | JavaScript / TypeScript（网页开发最常用的语言） |
| **手机上** | 通过 Capacitor 把网页包装成 iOS 应用 |
| **电脑上** | 通过 Electron 把网页包装成 Windows 应用 |

**👍 优点：**

1. **技术最普及**：网页开发是全世界最主流的编程方式，教程多到看不完
2. **生态最大**：想要什么功能几乎都有现成的"积木"可以拿来用
3. **界面灵活**：想做成什么样都能实现
4. **招人容易**：会网页开发的人最多，以后如果要找人继续开发很好找

**👎 缺点：**

1. **性能不如方案 A**：网页包装出来的应用，流畅度不如 Flutter 这种"原生"方案
2. **电脑应用体积很大**：Electron 打包出来的应用至少有 100MB+，因为它要内嵌一个整个浏览器（Chromium）
3. **体验有"网页感"**：操作手感可能不如真正的原生应用顺滑
4. **维护成本略高**：手机端和桌面端虽然共享大部分代码，但还是要分别处理一些平台差异

---

### 方案 C：纯网页应用（PWA）

**一句话解释**：不用下载安装，浏览器打开就能用，也可以添加到手机/电脑桌面，看起来像个 App。

| | 说明 |
|---|---|
| **用什么语言** | JavaScript / TypeScript |
| **手机上** | Safari 浏览器打开，添加到主屏幕 |
| **电脑上** | Chrome/Edge 浏览器打开，可"安装"到桌面 |

**👍 优点：**

1. **最简单最省钱**：不用学 App 开发那一套，也不需要苹果开发者账号（那个每年要交 $99/年）
2. **零安装**：发个链接给家人就能用，不用教怎么安装
3. **更新方便**：改动后刷新页面就行，用户永远是最新版本
4. **跨平台零成本**：不管什么系统，有浏览器就能用

**👎 缺点：**

1. **离线能力弱**：没网的时候基本用不了（虽然可以做一些缓存，但效果有限）
2. **不够"像 App"**：没有原生应用的滑动手势、系统通知等高级功能
3. **不能上架 App Store**：没法在苹果商店被搜索下载，只能通过链接分享
4. **数据存浏览器**：清除浏览器数据可能会丢失记账数据

|----------|:---:|:---:|:---:|

| 应用体积 | 中等 | 较大 | 无需安装 |
| 离线使用 | ✅ 完全支持 | ✅ 支持 | ⚠️ 有限 |
| 上架 App Store | ✅ 可以 | ✅ 可以 | ❌ 不行 |
| 需要苹果开发者账号 | ✅ 需要 | ✅ 需要 | ❌ 不需要 |
| 维护难度 | 低 | 中等 | 低 |
| 未来扩展能力 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |

## 五、项目开发约定

> ⚠️ **这是整个项目最重要的约定，请仔细阅读。**

### 5.1 关于技术决策

我是一个**技术小白**，不懂编程术语。因此在整个项目开发过程中，必须遵守以下规则：

1. **所有技术决策，必须由 AI（Claude Code）列出 2~3 个可选方案**
2. **每个方案都需要用通俗易懂的大白话解释**——是什么、有什么好处、有什么坏处
3. **由我来拍板决定选哪个方案**
4. **AI 不能替我擅自做技术决定**

### 5.2 关于沟通方式

1. 避免使用技术术语；如果必须使用，要附带通俗解释
2. 解释时尽量用类比（比如"这个就像……"）
3. 每一步操作前，用一句话告诉我要做什么以及为什么
4. 如果出现错误，告诉我发生了什么、严重不严重、下一步怎么办

### 5.3 关于文档维护

1. 本文档（`abc.md`）是项目的"总指挥部"，记录所有重要决策
2. 每次做完一个阶段的工作，更新本文档的进度记录
3. 技术方案选定后，将最终选择记录到本文档中

### 5.4 关于项目进度

- 每个阶段做完后，在本文档末尾的**项目日志**中记录
- 格式：日期 + 做了什么 + 有什么决定

### ⚠️ 关于 iOS（苹果手机）的重要说明

由于你用的是 Windows 电脑，我们**无法在这台电脑上直接打包 iOS 应用**。苹果要求必须用 Mac 电脑来生成 iPhone 应用。不过好消息是：

- **所有的代码**都可以在 Windows 上写，一份代码同时适用于 Windows 和 iOS
- **Windows 桌面版**可以在当前电脑上直接开发、运行、测试
- **iOS 版本**需要一个 Mac 电脑来打包发布（后续可以考虑借用朋友的 Mac，或使用云端 Mac 服务）
